# 代码模块功能讲解

本文档用于说明各模块的职责和设计思路。项目已完成全部代码实现，可直接运行。

## 1. `main.py`

项目入口文件。

主要职责：

- 导入游戏主类
- 调用游戏启动函数
- 保持入口简洁，避免把具体逻辑写在入口文件中

建议内容：

```python
from super_mario.game import Game

def main():
    game = Game()
    game.run()

if __name__ == "__main__":
    main()
```

## 2. `super_mario/config.py`

配置模块。

主要职责：

- 保存窗口宽高
- 保存帧率
- 保存瓦片大小
- 保存重力、跳跃速度、玩家速度
- 保存素材路径
- 保存常用颜色

这样做的好处：

- 参数集中管理
- 修改游戏速度或窗口大小时更方便
- 避免代码中到处出现难以理解的数字

## 3. `super_mario/level_data.py`

关卡数据模块。

主要职责：

- 使用字符串列表保存地图
- 用不同字符代表不同对象
- 为后续多关卡扩展提供基础

推荐符号：

| 字符 | 含义 |
|---|---|
| `#` | 地面或砖块 |
| `P` | 玩家出生点 |
| `C` | 金币 |
| `E` | 蘑菇敌人 |
| `B` | 飞行敌人 |
| `Q` | 问号砖块 |
| `U` | 已触发问号砖块 |
| `F` | 终点旗 |
| `.` | 空白 |

## 4. `super_mario/core.py`

核心规则模块。

主要职责：

- 定义玩家、敌人、金币、地图等数据结构
- 实现矩形碰撞检测
- 实现玩家重力和跳跃规则
- 实现金币收集规则
- 实现敌人踩踏和受伤规则
- 判断游戏是否通关或失败

推荐把 `core.py` 写成尽量不依赖 Pygame 的模块。这样可以单独测试规则，也能让老师看出程序逻辑不是完全依赖界面。

## 5. `super_mario/sprites.py`

精灵模块。

主要职责：

- 封装玩家精灵
- 封装敌人精灵
- 封装金币精灵
- 封装砖块精灵
- 处理图片加载、动画帧和绘制位置

如果项目规模较小，也可以暂时不写 `sprites.py`，把精灵绘制逻辑放在 `game.py` 中。但答辩时建议说明：当功能增加时，独立精灵模块更利于维护。

## 6. `super_mario/game.py`

游戏主控制模块。

主要职责：

- 初始化 Pygame
- 创建窗口
- 加载素材
- 处理键盘输入
- 调用核心规则更新游戏状态
- 绘制背景、地图、玩家、金币、敌人、UI
- 播放音乐和音效
- 控制开始、游戏中、通关、失败等状态

`game.py` 是连接“规则”和“画面”的模块。它不应该写太多底层规则，规则应尽量放在 `core.py`。

## 7. `assets/`

素材目录。

主要职责：

- 保存图片素材
- 保存音频素材
- 与代码分离，便于替换和管理

建议在代码中通过相对路径加载素材，避免写死个人电脑上的绝对路径。

## 8. 模块之间的关系

```text
main.py
  ↓
game.py
  ↓        ↘
core.py    sprites.py
  ↓
level_data.py
  ↓
config.py
assets/
```

解释：

- `main.py` 只负责启动。
- `game.py` 负责主循环、输入、绘制和音效。
- `core.py` 负责游戏规则 (纯 Python, 不依赖 Pygame)。
- `sprites.py` 负责 Pygame 图片加载和绘制。
- `level_data.py` 提供关卡文本地图。
- `config.py` 提供全局配置常量。
- `assets/` 提供图片和音频素材。

## 9. 注释要求建议

课程作业建议使用三类注释：

1. 文件顶部注释：说明这个模块做什么。
2. 类和函数注释：说明参数、返回值和功能。
3. 关键逻辑注释：说明碰撞检测、状态切换、分数计算等不容易一眼看懂的代码。

不建议每一行都写重复注释，例如 `x += 1` 不需要写“x 加一”。注释应解释“为什么这样做”，而不是机械翻译代码。
